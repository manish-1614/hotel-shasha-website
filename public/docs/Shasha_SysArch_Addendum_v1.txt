1.3 Booking Workflow — Step by Step

Step 1: Availability Check
Guest Action
Selects check-in / check-out dates on the availability calendar widget.
Frontend
Calls GET /api/availability?checkin=YYYY-MM-DD&amp;checkout=YYYY-MM-DD&amp;room=duplex-triple
API Route
Queries NeonDB blocked_dates + bookings tables. Returns: { available: true|false, price: number, nights: number }
UI Response
Calendar shows unavailable dates in red. Available dates highlighted in forest green. Price estimate shown instantly.

Step 2: Booking Creation (Pending)
Guest Action
Confirms room, dates, guests, meal plan. Clicks &apos;Book Now&apos;.
API: POST /api/book
Validates inputs via Zod. Checks availability again (double-check race condition guard). Creates booking row with status=&apos;pending&apos;. Locks dates in availability table for 15 minutes (TTL via Upstash KV).
Response
Returns { bookingId, orderId, totalAmount, depositAmount, expiresAt }

Step 3: Payment
Guest Action
Enters payment details in Razorpay checkout modal (SDK loaded client-side).
API: POST /api/payment/create
Creates Razorpay order via server-side SDK call. Amount = depositAmount (20–30% of total). Returns razorpay_order_id.
Razorpay SDK
Opens payment modal. Guest completes payment (UPI / card / net banking / wallet).
API: POST /api/payment/verify
Receives razorpay_payment_id + razorpay_signature. Verifies HMAC-SHA256 signature server-side. Updates booking status to &apos;confirmed&apos;. Releases availability lock. Writes confirmed dates to blocked_dates.
Notifications
Sends guest confirmation email + WhatsApp message. Sends host notification with booking summary.

Step 4: Webhook Handling (Async)
Razorpay Webhook
POST /api/webhooks/razorpay — handles: payment.captured, payment.failed, refund.created, order.paid
Signature verify
All webhooks verified via X-Razorpay-Signature header before processing.
payment.failed
Booking status → &apos;payment_failed&apos;. Availability lock released. Guest notified.
refund.created
Booking status → &apos;cancelled&apos;. Dates unblocked. Refund confirmation sent.

  🔄  SECTION 2: OTA SYNCHRONISATION STRATEGY  

2. OTA Synchronisation Strategy
Online Travel Agencies (OTAs) like Airbnb, Booking.com, and Hostelworld are critical distribution channels for Shasha — especially early on when direct booking traffic is low. Without synchronisation, any OTA booking risks causing a double-booking on the direct website, which is catastrophic for guest trust and host reputation.

2.1 The Double-Booking Problem
WITHOUT SYNC — The Danger Zone:  Airbnb Guest books Dec 20–22  →  Airbnb calendar updated  Website Guest books Dec 21–23  →  Website calendar NOT updated  ❌ DOUBLE BOOKING — host must cancel one, loses superhost statusWITH SYNC — The Safe Zone:  Airbnb Guest books Dec 20–22  →  iCal feed pushed to sync service  Sync Service  →  Updates blocked_dates in NeonDB  Website Calendar  →  Dec 20–22 now shows as unavailable  ✅ No double booking possible

2.2 OTA Sync Architecture

2.3 Phase 2A — iCal Sync (Low Cost, High Value)
iCal (RFC 5545) is the universal calendar standard supported by all major OTAs. It requires no paid channel manager — just two endpoints.

Outbound: Website → OTAs
The website exposes a read-only iCal feed per room type that OTAs subscribe to. This tells OTAs which dates are blocked from direct bookings.

// GET /api/ical/[roomSlug] — returns iCal feed// OTAs poll this URL every 15–60 minutes// Example: /api/ical/duplex-triple// /api/ical/private-room// /api/ical/dorm// Response: text/calendarBEGIN:VCALENDARVERSION:2.0PRODID:-//Shasha Jibhi//Booking//ENBEGIN:VEVENTDTSTART;VALUE=DATE:20251220DTEND;VALUE=DATE:20251223SUMMARY:BLOCKEDUID:sha-booking-uuid@shashajibhi.comEND:VEVENTEND:VCALENDAR

Inbound: OTAs → Website
The website imports and periodically syncs iCal feeds published by each OTA. When an Airbnb booking is made, the Airbnb iCal feed is updated — our sync job detects this and blocks those dates in NeonDB.

2.5 OTA Sync Data Flow Summary
OTA Booking
Guest books on Airbnb/Booking.com. OTA updates its internal calendar.
iCal Push
OTA publishes updated .ics feed at its export URL (15–60 min delay).
Cron Job
Vercel Cron (/api/cron/sync-ota) fires every 15 min. Fetches all OTA .ics feeds.
Diff Engine
Compares fetched events against blocked_dates in NeonDB. Identifies new blocks and cancellations.
DB Update
Upserts new blocked_dates rows. Removes rows for cancelled OTA bookings. Sets source=&apos;ota_airbnb&apos; etc.
Website Cal
Availability calendar reads from blocked_dates. Newly blocked dates immediately reflect as unavailable.
Race guard
If a direct booking is in-flight during a sync, the pessimistic lock (Upstash KV 15-min TTL) prevents a double-confirm.

5. Monitoring &amp; Logging
Production systems fail silently without monitoring. For Shasha, a broken contact form, a failed payment webhook, or a crashing SSR page could mean lost bookings with no alert to the host or developer. This section defines a comprehensive observability stack covering error monitoring, structured logging, and alerting.
